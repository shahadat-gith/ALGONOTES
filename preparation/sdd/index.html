Software Design Document

ALGONOTES Preparation Module

Version:1.0
Status:Implementation-Ready
Audience:Backend, Frontend, and AI Engineers implementing this module

1. Purpose and Scope

The Preparation Module is the core engine of ALGONOTES. It takes a user's resume and a target job description and produces a personalized interview preparation roadmap, broken into topics, each with an on-demand AI-generated discussion and a grounded follow-up chat.

This document specifies database schemas, API contracts, AI pipeline internals, prompt architecture, embedding strategy, caching, security, performance, and error handling at a level sufficient for direct implementation. It assumes the AI provider and embedding provider are abstracted behind interfaces (src/ai/provider.js, src/ai/embeddings.js) and is provider-agnostic by design.

Three domain modules (preparation, topic, chat) own their own routes/controller/service/model, and share a common ai/ layer for provider calls, embeddings, and retrieval, plus a prompts/ layer for prompt construction. This separation means the AI vendor can be swapped by editing only src/config/ai.js and src/ai/provider.js.

3. User Flow (Preparation Creation)

  1. Client submits company, role, resume (PDF, multipart), jobDescription (text) to POST /preparations.
  2. middlewares/upload.js (Multer, memory storage) validates the file and hands a buffer to the controller. The buffer is never written to disk and never persisted — it lives only for the duration of the request.
  3. preparation/controller.js creates a preparations document with status: "processing" and returns 202 Accepted with the preparationId immediately, so the client isn't blocked on AI calls.
  4. preparation/processor.js runs asynchronously (in-process job, see §13):
    • utils/parser.js extracts raw text from the PDF via pdf-parse.
    • AI call #1: structure the resume text into structuredResume.
    • AI call #2: structure the job description text into structuredJobDescription.
    • preparation/service.js persists resumeText, jobDescription (original text), structuredResume, structuredJobDescription to MongoDB. The PDF buffer is discarded — never written to disk or object storage.
  5. preparation/embeddings.js chunks the structured resume and JD section-by-section (see §10) and calls ai/embeddings.js to generate vectors, then upserts them into the preparation_context Qdrant collection.
  6. preparation/planner.js runs the Preparation Planner AI call: given the structured resume, structured JD, company, and role, it returns a topic list (title, category, priority, reason, estimatedTime) with no discussion content.
  7. topic/service.js bulk-inserts one topics document per planned topic, each with discussion: null.
  8. preparations.status transitions processing → ready (or failed with an error reason). The client polls GET /preparations/:id or receives a websocket/SSE event (polling is the baseline contract) until status === "ready", then fetches GET /preparations/:id/topics.

This flow guarantees: only extracted/derived text is stored, raw files are never persisted, and the expensive "topic list" generation happens once per preparation, not once per topic.

4. Topic Flow (Lazy Discussion Generation)

  1. Client calls GET /topics/:id.
  2. topic/controller.js → topic/service.js fetches the topic by id.
  3. If topic.discussion is non-null, return it immediately — no AI call, no Qdrant call. This is the dominant path after the first read.
  4. If topic.discussion is null:
    1. topic/service.js loads the parent preparation (for company, role, structured summaries).
    2. ai/retriever.js queries Qdrant filtered by preparationId, retrieving the top-k chunks most relevant to the topic's title + category + reason (used as the retrieval query), separately for type: "resume" and type: "jd", so both sides are represented.
    3. topic/generator.js builds the Topic Generator prompt (see §11) with: company, role, topic metadata, retrieved resume chunks, retrieved JD chunks.
    4. AI call produces the discussion (structured markdown/sections — see schema in §6).
    5. topic/service.js writes discussion, discussionVersion: 1, discussionGeneratedAt: now() back onto the topic document via a single atomic findOneAndUpdate.
    6. Return the discussion.

Regeneration only happens via an explicit force=true flag or a future "regenerate" endpoint — never implicitly. This is enforced in topic/service.js by checking discussion !== null && !options.force before even constructing the AI call, so a regeneration request can never be triggered accidentally by a normal page load.

5. Follow-up Chat Flow

  1. Client calls POST /topics/:id/chat with { message: string }.
  2. chat/controller.js validates the topic exists and (if discussion is null) returns 409 Conflict — a topic must be opened (discussion generated) before it can be chatted about, since the chat is grounded in that discussion.
  3. chat/service.js assembles grounded context:
    • The topic document (title, category, discussion).
    • Relevant resume chunks and JD chunks from Qdrant — re-retrieved per message using the user's new message as the query (not the original topic title), so follow-ups that drift into adjacent areas still pull the right chunks.
    • The prior conversation: the last N messages (default 10, configurable) from the topic's chats document, used as short-term memory.
  4. prompts/chat.js builds the Tutor Chat prompt (see §11) combining: base system prompt, discussion, retrieved chunks, conversation history, new user message.
  5. AI call returns the assistant's reply.
  6. chat/service.js appends both the user message and assistant reply to the chats document ($push with $each) and returns the assistant reply (and optionally the full updated message list, depending on ?include=history).

The chat never answers from unconstrained model knowledge — every call is required to pass discussion + chunks + history, enforced structurally because prompts/chat.js throws if discussion or chunks are missing, rather than silently producing an ungrounded prompt.

6. Database Design — MongoDB

6.1 preparations

Purpose: One document per "prepare me for Company X, Role Y" session. The root aggregate that topics and (transitively) chats hang off of.

Field Type Required Description
_id ObjectId auto Primary key.
userId ObjectId yes Owner, references the external ALGONOTES Auth user id. Trusts the auth service's id.
company String yes Free-text company name as entered by the user (e.g., "Stripe"). Used in prompts.
role String yes Free-text job role/title (e.g., "Backend Engineer II").
resumeText String yes Raw text extracted from the uploaded PDF via pdf-parse. Kept for re-chunking.
jobDescription String yes Raw JD text as pasted by the user.
structuredResume Object yes AI-extracted structured resume — see §6.1.1 sub-schema.
structuredJobDescription Object yes AI-extracted structured JD — see §6.1.2 sub-schema.
analysis Object no AI-generated comparison output (strengths, gaps, fit summary) — see §6.1.3.
embeddingStatus String enum yes One of pending, in_progress, completed, failed.
topicGenerationStatus String enum yes One of pending, in_progress, completed, failed.
status String enum yes Overall lifecycle: processing, ready, failed.
failureReason String no Human-readable error message, set only when status === "failed".
createdAt Date yes Set on insert.
updatedAt Date yes Updated on every write (Mongoose timestamps: true).

6.2 topics

Purpose: One document per planned preparation topic. Holds both the plan metadata and, once generated, the full discussion embedded in the same document.

Field Type Required Description
_id ObjectId auto Primary key.
preparationId ObjectId yes FK to preparations._id.
title String yes E.g., "Designing idempotent payment APIs".
category String enum yes One of dsa, system_design, behavioral, domain_knowledge, resume_deep_dive, company_specific.
priority String enum yes high, medium, low.
reason String yes Justification for including this topic. Reused in retrieval context.
estimatedTime Number yes Minutes, AI-estimated reading/prep time.
discussion Object | null no The generated content — see §6.2.1. null until first opened.
discussionVersion Number yes Defaults to 0. Increments on every regeneration.
discussionGeneratedAt Date | null no Timestamp of last generation.
completed Boolean yes Defaults to false. User-marked "done studying this".
completedAt Date | null no Set when completed flips to true.

6.2.1 discussion sub-schema

{
  summary: String,
  sections: [{
    heading: String,
    content: String
  }],
  keyPoints: [String],
  suggestedQuestions: [String]
}

Why embed discussion in the Topic document: A discussion is strictly 1:1 with a topic, and they are always read together. Embedding avoids an expensive join operation on the high-traffic GET /topics/:id path. Document sizes remain small (a few KB of markdown), well under MongoDB's 16MB threshold.

Indexes:

db.topics.createIndex({ preparationId: 1, priority: 1 });
db.topics.createIndex({ preparationId: 1, completed: 1 });

6.3 chats

Purpose: Conversation log for follow-up questions on a specific topic.

Field Type Required Description
_id ObjectId auto Primary key.
topicId ObjectId yes (unique) FK to topics._id. Enforces one chat document per topic.
messages Array yes Ordered list of { role, content, createdAt } objects.

Why one chat document per topic: Keeps chat logs isolated from the topic's structural metadata and discussion records. Reading the full conversation context requires just a single query instead of a complex paginated scan.

Indexes:

db.chats.createIndex({ topicId: 1 }, { unique: true });

7. Qdrant Design

7.1 Collection

Name: preparation_context
Vector size: Matches configured model dimension dynamically (read from config/ai.js).
Distance metric: Cosine similarity.

7.2 Payload schema

{
  preparationId: string,   // String-cast Mongo ObjectId used as the primary partition filter
  type: "resume" | "jd",
  section: string,         // e.g., "experience", "skills", "responsibilities"
  title: string,           // e.g., "Razorpay — SDE 2"
  text: string,            // Content chunk text passed back directly to the LLM context
  chunkIndex: number       // Position index within section for debugging and ordering
}

7.3 Chunking Strategy

Chunking occurs inside utils/chunk.js before hitting the embedding model and is section-aware. Individual logical elements (e.g., a single experience block or a bullet points group) represent individual chunks to maximize semantic boundaries. Token thresholds feature a sliding window fallback (300 tokens, 50 overlap) when sections overshoot max lengths.

7.4 Retrieval

async function retrieveContext({ preparationId, query, type, topK = 5 }) {
  const queryVector = await embed(query);
  return qdrant.search("preparation_context", {
    vector: queryVector,
    filter: {
      must: [
        { key: "preparationId", match: { value: preparationId } },
        ...(type ? [{ key: "type", match: { value: type } }] : [])
      ]
    },
    limit: topK
  });
}

9. API Design

All paths are prefixed with /api/v1. Requests require an Authorization: Bearer <token> header.

9.1 POST /preparations

Request Body: multipart/form-data containing company, role, jobDescription, and resume (PDF).

Response 202 Accepted:

{ "preparationId": "665f1a2b3c4d5e6f7a8b9c0d", "status": "processing" }

9.2 GET /preparations/:id/topics

Response 200 OK:

{
  "items": [
    { "id": "665f1b...", "title": "Designing idempotent payment APIs", "category": "system_design", "priority": "high", "reason": "...", "estimatedTime": 15, "completed": false, "hasDiscussion": false }
  ]
}

9.3 GET /topics/:id

Fetches or lazily evaluates a topic's markdown documentation on demand. Returns the full topic model instance.

Request Body: { "message": "Can you give a concrete example with a retry?" }

Response 200 OK:

{

}

10. AI Pipeline

  • Preparation Processor (processor.js): Extracts plain text from incoming file buffers via pdf-parse. Emits clean structured summaries of resumes and job briefs concurrently using parallel JSON generation contracts.
  • Embedding Generator (embeddings.js): Maps structural components to specific domain keys inside Qdrant collection spaces.
  • Preparation Planner (planner.js): Analyzes comprehensive data summaries directly to output optimized subject roadmaps containing clear priorities and rationales.
  • Topic Generator (generator.js): Generates detailed technical deep-dives utilizing targeted, contextual chunks instead of large complete document dumps.

11. Prompt System

11.1 Base Prompt (prompts/base.js)

"You are an expert technical interview coach embedded in ALGONOTES. You only use the information explicitly provided to you in the prompt (resume content, job description content, retrieved context chunks). You never invent specific facts about the candidate's background or the employer that are not present in the provided context. If asked something the provided context cannot answer, say so plainly rather than guessing."

12. Security and Guardrails

  • Memory Isolation: Raw multi-part document files handle streams inside server volatility buffers (memoryStorage). Buffers terminate immediately following text extraction execution loops.
  • Authorization Scoping: All database mutations and multi-tenant fetching constraints validate resource references against verified request signatures (req.userId).
  • Prompt Injection Protections: External user input surfaces get explicitly encapsulated within data blocks bounded by delimiter tags (e.g., <<<RESUME>>>) to avoid execution command overrides.

13. Error Handling and Recovery

  • Empty Vector Protections: Text extractions evaluating to less than 50 characters fail gracefully, assigning a clean ParsingError status block to failureReason attributes.
  • JSON Format Repairs: Failed structured schema evaluations trigger an explicit single-iteration formatting repair loop displaying diagnostic errors directly to the generation model.
  • Atomic Operation Safety: State updates execute changes utilizing targeted findOneAndUpdate instructions, preventing partial transaction writes.
